## 一、你现在掌握的分布式核心能力

1. **服务无状态化**

    * 将本地静态文件迁移到 MinIO，实现 Pod 无状态化
    * ✅ 可以水平扩容，Pod 可替换，不依赖宿主机

2. **共享资源初始化的分布式锁**

    * 使用 Redis 分布式锁控制多副本初始化顺序
    * ✅ 理解多副本同时操作共享资源会导致冲突

3. **健康检查（Service Discovery / Liveness & Readiness）**

    * HTTP `/health` 接口让 K8s 判断容器健康
    * Pulsar 消费健康判断，间接判断任务消费状态
    * 定时任务刷新文件 + K8s 脚本判断
    * ✅ 能实现**多副本服务的健康监控**，包括应用级和任务级健康

4. **日志与监控**

    * Prometheus SDK 采集指标
    * Grafana 展示可视化面板
    * ✅ 能跟踪关键接口调用和任务执行状态，做基础监控

5. **调试策略**

    * 日常调试多副本环境，降副本到 1 便于查看日志
    * ✅ 理解多副本日志分散问题

## 二、你理解的分布式短板 / 待提升点

| 短板点             | 说明                                                        |
|-----------------|-----------------------------------------------------------|
| **多副本流量调度**     | 你知道多副本存在，但对流量如何分发、负载均衡策略（K8s Service、Ingress、Gateway）理解较浅 |
| **分布式一致性**      | Redis 分布式锁只是解决初始化问题，你对**跨节点数据一致性、幂等性、分布式事务、最终一致性**理解不足    |
| **服务发现深度**      | 你实现了健康检查，但对完整的服务发现机制（如自动注册/注销、DNS/Env 发现、动态负载感知）未系统掌握     |
| **日志集中化和可观测性**  | 你使用 Grafana 可视化，但对**日志聚合、追踪链路、Alert/告警策略**了解有限            |
| **容错和弹性**       | 健康检查只是被动感知 Pod 状态，你对**副本失败重试、Leader 选举、网络分区处理、熔断降级**未深度涉及 |
| **调度与扩缩容策略**    | 对 HPA/Cluster Autoscaler 的原理、Pod 调度策略、节点亲和性、Pod 亲和性理解不够   |
| **分布式消息/任务的容错** | Pulsar 消费健康判断是经验性做法，但对**消息重试、幂等消费、消息顺序保证**等高级分布式模式了解不足    |
